버튼, 텍스트 입력, 테이블, 프로그레스 바 등 UI 부품을 제공합니다.

CMake C++ 프로젝트의 빌드 설정 도구.
"어떤 소스 파일을 어떤 라이브러리와 결합해서 어떤 실행 파일을 만들지"를 정의합니다.
CMakeLists.txt 파일에 설정을 작성합니다. 		CLI용 CMakeLists.txt: GEANT4 + 소스코드 → ShieldDose 바이너리
GUI용 CMakeLists.txt: Qt5 + GUI 소스코드 → ShieldDoseGUI 바이너리
Makefile CMake의 구식 대안. make 명령어로 빌드를 실행합니다.
CMake가 더 현대적이고 크로스 플랫폼을 지원하여 신규 프로젝트에서 선호됩니다. 		기존 GUI (Phase 2)에서 사용했으나, Phase 3에서는 CMake로 전환.
프로세스
Process		 실행 중인 프로그램. OS가 메모리를 할당하고 관리하는 단위.
하나의 프로그램에서 다른 프로그램을 "프로세스 생성(spawn)"하여 실행할 수 있습니다. 		GUI 프로세스가 CLI 프로세스를 생성(spawn)하여 시뮬레이션 실행.
두 프로세스는 독립적 — GUI가 죽어도 시뮬레이션은 계속됩니다.
Monte Carlo
MC 시뮬레이션		 난수(랜덤)를 사용해 물리 과정을 통계적으로 모사하는 방법.
입자 하나하나의 궤적을 추적하여, 수만~수백만 개의 입자를 시뮬레이션합니다.
이벤트 수가 많을수록 결과가 정밀해집니다. 		GEANT4 = Monte Carlo 시뮬레이션 엔진.
ShieldDose에서 보통 50만~100만 이벤트를 실행합니다.
CSV Comma-Separated Values. 쉼표로 구분된 텍스트 데이터 파일.
엑셀에서 열 수 있는 간단한 표 형식입니다. 		GEANT4가 시뮬레이션 결과를 CSV로 출력.
GUI가 이 CSV를 읽어서 시각화합니다.
STL 파일 3D 형상을 삼각형 메시로 표현한 파일 포맷.
CAD 프로그램(Onshape, SolidWorks 등)에서 내보내기 가능. 		Onshape에서 차폐체 형상을 STL로 내보내 → ShieldDose에 입력.

03GEANT4란 무엇인가

GEANT4 = GEometry ANd Tracking, version 4

CERN(유럽입자물리연구소)이 개발한 C++ 입자 물리 시뮬레이션 프레임워크.
1998년 첫 배포 이래, 전 세계 수천 개 연구기관에서 사용 중.
고에너지 물리, 의학물리, 우주방사선, 방사선 방호 등에 활용.

GEANT4가 하는 일

GEANT4는 입자 하나하나의 여행(궤적)을 물리 법칙에 따라 추적합니다:

감마선 광자 발생 (선원에서)
    │
    ▼ 직진 이동 (랜덤 거리)
    │
    ├─ Compton 산란 → 방향 변경 + 에너지 감소 → 계속 이동
    │
    ├─ 광전 효과  → 광자 흡수 → 전자 방출 → 추적 종료
    │
    ├─ 쌍생성     → e⁺ + e⁻ 생성 (>1.022 MeV)
    │
    └─ 디텍터 도달 → 에너지 침적(deposit) 기록 → 선량 계산

이 과정을 50만 번 반복하면 통계적으로 의미 있는 결과를 얻습니다. 이것이 Monte Carlo 방법입니다.

GEANT4가 제공하는 것

물리 모델

전자기 상호작용, 핵반응, 붕괴 등 수백 가지 물리 과정의 수학적 모델. CERN이 수십 년간 실험으로 검증.

입자 추적 엔진

3D 공간에서 입자의 이동, 상호작용, 에너지 손실을 단계별로 계산하는 핵심 엔진.

기하 시스템

상자, 원통, 구 등 기본 형상과 STL 메시 등을 조합하여 시뮬레이션 공간을 구성.

물질 데이터베이스

G4_Pb(납), G4_WATER(물), G4_CONCRETE(콘크리트) 등 수백 종의 물질과 그 물리적 특성.

중요: GEANT4는 라이브러리입니다. 단독으로 실행할 수 없습니다.
반드시 사용자가 C++ 프로그램을 작성하여 GEANT4 라이브러리를 호출해야 합니다.
이 "사용자가 작성하는 C++ 프로그램"이 바로 ShieldDose CLI입니다.

04일반적인 GEANT4 사용법

전 세계 연구자들이 GEANT4를 사용하는 일반적인 과정입니다. 이 과정이 얼마나 복잡한지 이해하면, 우리 GUI의 가치가 명확해집니다.

Step 1: C++ 코드 작성 (연구자가 직접)

GEANT4를 사용하려면 연구자가 최소 5개의 C++ 클래스를 작성해야 합니다:

연구자가 작성해야 하는 파일들:

1. DetectorConstruction.cc  — "시뮬레이션 공간(기하)" 정의
   │  "납 10cm 두께 벽 + 물 디텍터를 이렇게 배치해라"
   │  C++로 직접 좌표를 계산하고 물체를 배치

2. PrimaryGeneratorAction.cc — "어떤 입자를 쏠 것인가" 정의
   │  "Co-60 감마선을 등방(모든 방향)으로 쏴라"
   │  에너지 스펙트럼, 방향, 위치를 코드로 설정

3. PhysicsList.cc           — "어떤 물리 과정을 켤 것인가" 선택
   │  "전자기 상호작용 option4 + 방사성 붕괴"
   │  잘못 선택하면 결과가 완전히 달라짐

4. SteppingAction.cc        — "입자가 이동할 때마다 무엇을 기록할 것인가"
   │  "디텍터에 에너지가 침적되면 그 값을 저장해라"

5. RunAction.cc             — "시뮬레이션 시작/종료 시 무엇을 할 것인가"
   │  "결과를 CSV로 저장하고 통계를 계산해라"

+ main.cc, EventAction.cc, ActionInitialization.cc 등...

Step 2: CMake로 빌드

# CMakeLists.txt 작성
cmake_minimum_required(VERSION 3.16)
project(MySimulation)
find_package(Geant4 REQUIRED)
add_executable(MySimulation main.cc src/*.cc)
target_link_libraries(MySimulation ${Geant4_LIBRARIES})

# 터미널에서 빌드
$ mkdir build && cd build
$ cmake ..
$ make -j4

Step 3: 매크로 파일 작성

# run.mac — GEANT4 매크로 파일
/run/initialize

# 선원 설정 (Co-60 감마선)
/gps/particle gamma
/gps/ene/type Mono
/gps/ene/mono 1.173 MeV
/gps/ang/type iso

# 50만 이벤트 실행
/run/beamOn 500000

Step 4: 실행 & 결과 분석

# 터미널에서 실행
$ ./MySimulation run.mac

# 결과 CSV 분석 (Python 스크립트 직접 작성)
$ python3 analyze_dose.py output.csv
일반 연구자의 GEANT4 진입장벽:
  • C++ 프로그래밍 능력 필수
  • GEANT4 API 학습 (방대한 클래스 체계)
  • CMake 빌드 시스템 이해
  • 물리 리스트 선택 지식 (잘못 고르면 결과 오류)
  • 결과 분석 스크립트 직접 작성
  • 디버깅 어려움 (잘못된 기하 → 침묵 실패)

ShieldDose GUI가 바꾸는 것

05ShieldDose CLI

ShieldDose CLI는 "일반 GEANT4 사용자가 직접 작성해야 하는 C++ 코드"를 방사선 방호 목적에 최적화하여 미리 작성한 것입니다.

ShieldDose CLI = 우리가 이미 완성한 GEANT4 애플리케이션

  일반 연구자가 하는 일          ShieldDose가 대신 해주는 일
  ─────────────────────         ────────────────────────────
  DetectorConstruction 작성  →  납/철/콘크리트 차폐 + 물 디텍터 자동 구성
  PrimaryGenerator 작성      →  6핵종(Co-60~Tc-99m) GPS 소스 매크로 내장
  PhysicsList 선택           →  최적 물리 리스트 선정 완료 (option4)
  SteppingAction 작성        →  에너지 침적 스코어링 구현 완료
  RunAction 작성             →  CSV 자동 출력 + 통계 계산 구현 완료
  분산감소 기법 구현          →  Importance Biasing 구현 완료

검증 상태:
  감마상수 6/6 핵종 ±10% PASS (P2.3wc~P2.5wc)
  HVL 6/6 핵종 PASS (P2.6wc)
  MC 독립 HVL 결정 완료 (38회 두께 스캔)

CLI 사용법

06GEANT4 핵심 컴포넌트 상세 해설

ShieldDose CLI 내부에서 사용되는 GEANT4 컴포넌트를 하나씩 설명합니다.

G4RunManager — GEANT4 커널 (총사령관) 
G4RunManager — GEANT4의 "운영체제"

  역할: 시뮬레이션의 모든 것을 총괄 관리
  ┌─────────────────────────────────────────┐
  │  G4RunManager                           │
  │  ├── 기하 구성 초기화                    │
  │  ├── 물리 리스트 초기화                  │
  │  ├── 이벤트 루프 실행                    │
  │  │   ├── Event 1: 입자 생성 → 추적 → 스코어링  │
  │  │   ├── Event 2: 입자 생성 → 추적 → 스코어링  │
  │  │   ├── ...                            │
  │  │   └── Event N: 입자 생성 → 추적 → 스코어링  │
  │  └── 결과 수집 및 출력                   │
  └─────────────────────────────────────────┘

프로그램 시작 시 G4RunManager가 생성되고, 모든 컴포넌트를 등록받아 시뮬레이션을 실행합니다. /run/beamOn 500000 명령은 "이벤트 50만 개를 실행하라"는 뜻입니다.

G4EmStandardPhysics_option4 — 물리 리스트 (물리 법칙 선택) 
물리 리스트 = "어떤 물리 법칙을 켤 것인가"

  GEANT4는 수백 가지 물리 과정을 제공합니다.
  모두 켜면 너무 느려지고, 잘못 고르면 결과가 틀립니다.

  option4 = 감마선/전자 시뮬레이션에 최적화된 설정
  ┌──────────────────────────────────────┐
  │  켜져 있는 물리 과정:                  │
  │  ✓ Compton 산란     (감마 → 감마 + e⁻)│
  │  ✓ 광전 효과        (감마 → e⁻)       │
  │  ✓ 쌍생성           (감마 → e⁺ + e⁻)  │
  │  ✓ Rayleigh 산란    (감마 → 감마)     │
  │  ✓ 전자 이온화      (e⁻ → e⁻ + e⁻)   │
  │  ✓ 제동복사         (e⁻ → e⁻ + 감마)  │
  │  ✓ 방사성 붕괴                        │
  └──────────────────────────────────────┘

  왜 option4?
  option1~3보다 낮은 에너지에서 더 정확함.
  방사선 방호에서는 수 keV~수 MeV 범위가 중요하므로
  option4가 최적의 선택입니다.
G4GeneralParticleSource (GPS) — 입자 소스 (방사선 선원) 
GPS = "어떤 방사선을 어디서 어떻게 쏘는가"

  3가지 설정 요소:

  1. 입자 종류    /gps/particle gamma
     감마선, 전자, 중성자, 양성자 등

  2. 에너지 스펙트럼
     ┌── Mono (단일 에너지)
     │   /gps/ene/type Mono
     │   /gps/ene/mono 0.662 MeV        ← Cs-137
     │
     ├── User Histogram (다중 에너지)
     │   /gps/ene/type User
     │   /gps/hist/point 0.2955 0
     │   /gps/hist/point 0.2965 28.7    ← 296 keV (28.7%)
     │   /gps/hist/point 0.3075 0       ← Ir-192 등
     │
     └── Multiple Vertex (캐스케이드)
         /gps/source/multiplevertex true ← Co-60
         1.173 MeV + 1.332 MeV 동시 방출

  3. 각도 분포    /gps/ang/type iso
     iso = 등방 (모든 방향 균등)
     → 실제 선원처럼 360도 모든 방향으로 방출

G4Box, G4PVPlacement — 기하 구성 (시뮬레이션 공간) 
시뮬레이션 공간 = 3D 상자 안에 물체 배치

  ┌───────────────── World (3m × 3m × 3m, 공기) ─────────────────┐
  │                                                              │
  │       ★ 선원 (0,0,0)                                       │
  │       │                                                      │
  │       │  (감마선 등방 방출)                                │
  │       ▼                                                      │
  │  ┌─────────────┐                                             │
  │  │  차폐벽 (Shield)│  ← G4Box + G4_Pb + 두께 설정            │
  │  │  100×100×T cm │  ← /shield/thickness T cm               │
  │  └─────────────┘                                             │
  │       │                                                      │
  │       │  (투과한 감마선)                                  │
  │       ▼                                                      │
  │  ┌─────────────┐                                             │
  │  │ 디텍터 (Detector)│ ← G4Box + G4_WATER + 크기 설정         │
  │  │  20×20×5 cm  │ ← 에너지 침적 → 선량 계산                  │
  │  └─────────────┘                                             │
  │                                                              │
  └──────────────────────────────────────────────────────────────┘

  G4Box       = 직육면체 형상 (가로 × 세로 × 높이)
  G4PVPlacement = 물체를 3D 공간의 특정 좌표에 배치
G4PSDoseDeposit — 스코어링 (선량 측정) 
스코어링 = "디텍터에서 무엇을 측정하는가"

  입자가 디텍터를 통과하면서 에너지를 남김 (에너지 침적)

  G4PSEnergyDeposit — 에너지 침적 [MeV]
  │  입자가 물질과 상호작용하며 전달한 에너지의 합
  │
  G4PSDoseDeposit   — 흡수 선량 [Gy]
  │  Dose = ΣEdep / (밀도 × 부피)
  │  단위: Gy (Gray) = J/kg
  │
  G4PSFlatSurfaceFlux — 표면 플럭스 (참고용)
     디텍터 표면을 통과하는 감마선 수

  선량 → 선량률 변환:
  DoseRate [Gy/h] = MeanDose [Gy/event] × Activity [Bq] × 3600 [s/h]

  예: MeanDose = 1×10⁻¹⁶ Gy, Activity = 37 GBq (1 Ci)
     → DoseRate = 1×10⁻¹⁶ × 3.7×10¹⁰ × 3600 = 13.3 mGy/h
G4ImportanceBiasing — 분산감소 (효율 향상 기법) 
문제: 두꺼운 차폐벽을 통과하는 감마선은 극소수

  50만 이벤트 중 차폐벽 뒤 디텍터에 도달하는 감마선:
  차폐 없음  → ~6,500개 도달 (1.3%)
  5cm Pb    → ~10개 도달 (0.002%)  ← 통계적으로 무의미!

해결: Importance Biasing (중요도 편향)

  차폐벽을 여러 층으로 나누고, 깊은 층에 들어갈수록
  입자를 "분열"시켜 통계를 인위적으로 늘림:

  ┌─────┬─────┬─────┬─────┬─────┐
  │ Pb  │ Pb  │ Pb  │ Pb  │ Pb  │  ← 차폐벽을 5층으로 분할
  │ ×1  │ ×2  │ ×4  │ ×8  │ ×16 │  ← 각 층의 "중요도"
  └─────┴─────┴─────┴─────┴─────┘

  입자가 ×2 → ×4 경계를 넘으면 2개로 복제 (가중치 반감)
  → 깊은 곳에서도 충분한 통계를 확보
  → 가중치 보정으로 물리적 정확도 유지

  결과:
  상대오차 99.9995% → <5% (P2.3wc에서 구현)

09빌드 시스템

빌드(Build)란 사람이 읽는 소스코드(C++)를 컴퓨터가 실행할 수 있는 바이너리로 변환하는 과정입니다.

소스코드              컴파일러           바이너리
main.cc          ─→  clang++       ─→  ShieldDose
src/*.cc              (Apple M5)        (실행 파일)
  +
GEANT4 라이브러리  ─→  링커          ─→  (GEANT4 코드 포함)

CMake의 역할

CMake는 이 변환 과정의 "레시피"를 정의합니다:

# CMakeLists.txt — "이 재료로 이 요리를 만들어라"

# 필요한 재료 (라이브러리) 찾기
find_package(Geant4 REQUIRED)     # GEANT4 라이브러리 위치 찾기

# 완성품 (실행 파일) 정의
add_executable(ShieldDose         # 이름: ShieldDose
    main.cc                       # 재료 1: 메인 코드
    src/DetectorConstruction.cc   # 재료 2: 기하 구성
    src/RunAction.cc              # 재료 3: 실행 동작
    ...                           # 나머지 소스 파일
)

# 라이브러리 연결 (링크)
target_link_libraries(ShieldDose
    ${Geant4_LIBRARIES}           # GEANT4와 결합
)

두 개의 독립된 빌드

CLI 빌드 (GEANT4 필요)

cd build/
cmake ..
make -j4
→ ShieldDose (CLI 바이너리)

GEANT4 + 소스코드 → 시뮬레이션 실행 파일

GUI 빌드 (Qt5만 필요)

cd ShieldDose_GUI_v3/build/
cmake ..
make -j4
→ ShieldDoseGUI (GUI 바이너리)

Qt5 + GUI 소스코드 → 사용자 인터페이스 실행 파일

왜 분리?
CLI 빌드에는 GEANT4 설치가 필수 (수 GB, 복잡한 의존성).
GUI 빌드에는 Qt5만 필요 (brew install qt@5, 간단).
분리하면 GUI 개발 시 GEANT4 빌드를 기다릴 필요 없음.

10왜 이렇게 설계하는가

핵심 질문: "GUI에 GEANT4를 직접 넣으면 안 되나?"

직접 임베드 (Phase 2 방식)

  • GUI + GEANT4가 같은 프로세스에서 실행
  • 시뮬레이션 중 GUI 멈춤 (이벤트 루프 충돌)
  • GEANT4 크래시 → GUI도 같이 죽음
  • GUI 코드 수정 → 물리 검증 재필요
  • 빌드 복잡 (Qt5 + GEANT4 동시 링크)
  • 결과: 25개 결함, 실질적 미작동

프로세스 분리 (Phase 3 방식)

  • GUI와 CLI가 별도 프로세스
  • 시뮬레이션 중 GUI 정상 반응 (비동기)
  • CLI 크래시 → GUI에서 에러 표시 (안전)
  • GUI 수정해도 물리 엔진 불변 (검증 유지)
  • 빌드 단순 (각각 독립 빌드)
  • 검증된 CLI 그대로 활용

실제 사례: 이 방식을 쓰는 다른 프로젝트들

프로젝트방식설명
TOPAS (MC 시뮬레이터) 매크로 파일 기반 GEANT4를 감싸서 간단한 텍스트 설정으로 사용. 세계적으로 가장 널리 쓰이는 의학물리 MC 도구.
GATE (의학영상 MC) 스크립트 기반 GEANT4 위에 매크로 레이어를 추가. PET/SPECT 시뮬레이션에 표준.
FLUKA 입력 카드 기반 CERN의 또 다른 MC 코드. 텍스트 입력 파일로 시뮬레이션 정의.
